# PR-D6e: ChatResponse `cached: bool` field

## Why this slice exists

Item #1 in the post-D6b LLM Gateway follow-up queue (locked 2026-05-05):

> Add a new boolean field on `ChatResponse` so customers can tell at
> a glance whether their request hit the cache (currently they have
> to infer from `usage = 0/0/0`, which is ambiguous if they actually
> sent zero-token input).

PR-D6b shipped the exact-cache wiring on `/api/v1/llm/chat`; on a hit,
the response carries `usage = ChatUsage(0, 0, 0)`. Customers can
detect cache hits today by inspecting that, but the signal is
ambiguous: a real provider call with zero input tokens (degenerate
but legal) looks identical.

Existing test
`tests/test_llm_gateway_router.py::test_chat_cache_hit_returns_zero_token_usage`
explicitly notes this as a follow-up:

```python
"""On cache hit: ChatResponse.usage = 0/0/0 (no tokens consumed
this call). Customer can infer cache via the zero usage; a
follow-up PR adds a ``cached: bool`` field for explicit signaling."""
```

This PR is that follow-up.

## Scope (this PR)

Three lines of source change in `atlas_brain/api/llm_gateway.py`:

1. Add `cached: bool = False` to the `ChatResponse` Pydantic model.
2. Cache-hit path returns `ChatResponse(..., cached=True)`.
3. Provider-call (miss) path returns `ChatResponse(..., cached=False)`.

Plus regression tests asserting both branches set the field
correctly.

## API contract change

**Additive only.** The new field defaults to `False`, so existing
clients that ignore it are unaffected. Pydantic deserializers ignore
extra fields by default; strict typed clients see a new optional
field with a default. No breaking change to any wire format.

## Intentional (looks wrong but is deliberate)

- **`cached: bool = False`** rather than `cached: Optional[bool] = None`.
  The field has a definite answer for every response (hit or miss
  is binary). `None` would imply "I don't know" -- never the truth
  for a `/chat` response.
- **Explicit `cached=False` in the miss-path return** rather than
  relying on the default. Symmetry with the hit-path explicit
  `cached=True` makes the contract visible at the call site; reduces
  drift risk if the model default changes later.
- **No corresponding field on `chat_stream` ChatStreamingResponse**.
  Item #3 in the post-D6b queue (apply exact cache to `/chat/stream`)
  is the right place for that -- streaming + cache has subtle
  semantics (single SSE event vs. chunked replay) that haven't been
  resolved. This PR scopes to the synchronous `/chat` route only.
- **No `cached` field on the cache-hit `llm_usage` row metadata.**
  The metadata already has `cache_hit: True` from PR-D6b -- adding
  `cached: True` would duplicate the signal under two names.

## Deferred (looks missing but is on purpose)

- Items #2-#6 in the post-D6b queue (cache savings dashboard,
  /chat/stream cache, cache opt-out header, semantic cache, /batch
  cache). #2 was already done in PR-D6c, but the others remain
  queued. PR-D6e is intentionally minimum-scope.
- Documentation surface (e.g., OpenAPI schema, customer docs).
  Pydantic's autogenerated schema will pick up the new field; the
  product-docs surface lives outside atlas and updates separately.
