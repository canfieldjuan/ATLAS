# PR-Deflection-Full-Report-QA-PDF-Text-Extraction

## Why this slice exists

#1612's ATLAS-side live runner is merged, but it still cannot produce the
actual live proof artifact without trusting a separately supplied PDF text file.
That was acceptable for #1625 because production delivers the PDF as an email
attachment, not an API download. It is not enough for the next slice, where the
proof needs to say the same PDF bytes being checked for leaks are also the bytes
whose text/counts were scored.

Root cause: the live runner split the PDF into two unrelated operator inputs:
`--pdf-bytes` for byte validity/leak scans and `--pdf-text` for scorecard
readability/count checks. The byte side can fail leaks, but the text side is
operator-asserted and can drift from the PDF. This PR fixes the root at the
runner seam by extracting text from the supplied PDF bytes by default. The
operator text file remains an explicit override for emergency/debug use, but the
normal proof path becomes byte-derived and records that fact in the sanitized
result.

Implementation found one deeper root that the byte-derived path exposed: the
PDF/export validator treated any occurrence of an end-marker phrase as a section
boundary. The rendered PDF repeats "complete evidence export" inside question
detail body copy, so the validator truncated the detail section before later
questions. This PR fixes that at the validator boundary by requiring section
markers to match exact heading lines, not arbitrary body substrings.

Review found a second proof-integrity root: raw PDF-byte leak scanning could not
see sensitive strings inside compressed fpdf2 content streams, so leak coverage
still depended on the text extractor. This PR moves that guard upstream by
scanning decompressed PDF streams for sensitive patterns before live network
calls, and by rejecting unsupported stream filters instead of treating undecoded
bytes as verified text.

This deliberately comes before the controlled live execution artifact. Running
the live proof while the text is still manually paired would recreate the exact
"harness around a deferred validation step" problem the root-cause gate is meant
to stop.

The diff is slightly over the soft cap because the runner extraction path and
the validator section-boundary fix are coupled: the first byte-derived proof
exposed the second defect, and shipping either side without its regression test
would leave the live artifact proof brittle.

## Scope (this PR)

Ownership lane: content-ops/deflection-full-report-qa
Slice phase: Functional validation

1. Make `--pdf-text` optional for the live full-report QA runner.
2. Extract text from supplied PDF bytes when `--pdf-text` is absent, using the
   decoded text streams emitted by the existing fpdf2 renderer.
3. Mark scorecard input text as `extracted_from_pdf_bytes` and
   `verified_from_pdf_bytes: true` when extraction succeeds.
4. Keep the existing `--pdf-text` operator override for manual comparison and
   mark that path as `operator_asserted`.
5. Scan decompressed PDF streams for sensitive patterns before network calls so
   leak coverage is independent of the text extractor.
6. Fail closed before network calls when byte-derived extraction returns empty
   text or cannot decode a text stream.
7. Tighten PDF/export section splitting so body copy mentioning "Complete
   Evidence export" does not terminate `Question Details and Evidence`.
8. Add focused tests using actual `render_deflection_full_report_pdf` bytes, a
   rendered compressed-stream leak, an unsupported-stream failure, a non-PDF
   extraction failure, and the legacy operator-text override.

## Mechanism

The runner will read the PDF bytes first and run the existing byte leak scan.
If `--pdf-text` is supplied, it keeps the current path. If not, it extracts text
from the bytes by scanning PDF streams, decompressing FlateDecode streams, and
decoding the literal strings used by fpdf2 text drawing operations. Sensitive
pattern checks run over both raw bytes and decompressed stream bytes before live
network calls. The extractor returns plain text only; it never writes raw
streams or PDF snippets to the result payload.

The extraction path is intentionally narrow and fail-closed. It targets the
production renderer we own today. If a future renderer emits an unsupported
encoding or an image-only PDF, the runner returns `ok: false` before network
calls instead of accepting an empty scorecard input.

The validator section splitter now finds end markers only on exact heading
lines instead of arbitrary substrings. That preserves the intended section
boundaries when ordinary body copy points readers to the complete evidence
export.

## Intentional

- No `pypdf` or `pdfminer` dependency yet. The current production renderer is
  fpdf2, and a small byte-stream extractor is enough to unblock the live proof
  without adding a new package surface.
- No OCR or image-only PDF support. A PDF whose text cannot be recovered fails
  closed, which is correct for a proof runner.
- The operator `--pdf-text` path remains available, but it is explicitly marked
  as not byte-verified in the output.
- Renderer-backed tests import the PDF renderer only inside tests that need
  fpdf2. This keeps the broad extracted-checks suite from failing collection
  when it runs without full PDF-renderer dependencies, while the report workflow
  still runs those tests with `requirements.txt`.
