# PR-Maturity-Sweep-Report-Validation-Tighten

## Why this slice exists

PR #1731 added a read-only maturity-sweep baseline debt report and the reviewer
left skip-worthy NITs on malformed inputs that should fail closed if the report
contract says malformed baseline JSON is rejected.

Root cause: the report validator reused Python's broad `int` type check, where
`bool` is an `int`, and it supplied permissive defaults for omitted `score` and
negative `--top-counts` values. That made malformed operator/report inputs look
valid instead of surfacing drift.

This fixes the root for the report tool by tightening validation at the parser
boundary, not by patching rendered output after malformed values have already
entered the summary.

## Scope (this PR)

Ownership lane: ci/maturity-sweep
Slice phase: Production hardening

1. Reject boolean values anywhere an integer baseline field is required.
2. Require each baseline entry to include an integer `score` field.
3. Reject negative `--top-counts` values instead of silently clamping them.
4. Add focused negative tests for the tightened failure branches.

### Review Contract

Acceptance criteria:
- Baseline `score` and `counts` values accept integers but reject booleans.
- Missing `score` fails with a clear baseline error.
- Missing `counts` remains accepted as an empty finding map because older
  generated baselines may omit zero-count maps.
- Negative `--top-counts` fails before report rendering.
- Existing summary ordering, JSON output, and no-match failure behavior remain
  unchanged.

Affected surfaces:
- Operator-only maturity-sweep baseline reporting.

Risk areas:
- Accidentally rejecting committed baselines generated by the existing
  `maturity_sweep.py` producer.
- Weakening the report-only nature of the tool by adding writes or gate logic.

- Reviewer rules triggered: R2, R10, R14.

## Mechanism

`_int_field` now treats `bool` as malformed even though Python models it as an
`int` subclass. `summarize_baseline` checks that every entry carries `score`
before reading it, while retaining the existing empty-map default for omitted
`counts`. `main` validates `--top-counts >= 0` before collection/rendering so a
bad CLI value exits non-zero with the same `maturity baseline summary:` prefix
as baseline parse failures.

## Intentional

- Missing `counts` still means an empty finding map. The real producer writes
  count maps, but accepting old zero-count baselines keeps this hardening
  backward-compatible without hiding score debt.
- This slice does not turn the report into a gate; it only makes the report's
  malformed-input handling match the contract claimed by #1731.
