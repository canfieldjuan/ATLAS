# PR-Content-Ops-Claim-Evidence-Artifact-Parser

## Why this slice exists

#1435 now has the deterministic benchmark path through artifact-directory
output: fixture validation, prompt/schema capture, injected-provider runner,
result artifact shaping, writeup rendering, JSON/Markdown bundling, safe
directory writes, and the #1509 CLI that writes the bundle from recorded
responses.

The remaining #1435 backlog is mostly live-provider execution and verifier
inclusion, but those are intentionally gated on real benchmark data. Before
that step, operators need one more deterministic utility: validate a saved
`claim_evidence_result.json` artifact and rehydrate it into the same package
contract that renders Markdown and drives go/no-go decisions. Without that,
saved result files are write-only evidence and future CI/operator checks would
need to either rerun model witnesses or trust raw JSON blindly.

This slice is over the normal 400 LOC target because the parser's shape
validation and negative fixtures must ship together; a parser without its
fail-closed branches pinned would be worse than leaving saved artifacts
write-only.

## Scope (this PR)

Ownership lane: content-ops/review-contract
Slice phase: Functional validation

1. Add a package-owned parser for saved claim/evidence result artifact JSON.
2. Rehydrate only the existing result dataclasses used by the renderer and
   writer.
3. Fail closed on malformed JSON, missing or wrong-shaped result sections,
   wrong scalar types, and contradictory derived verdict, `ok`, or
   `go_no_go` fields.
4. Keep live providers, prompt execution, verifier rubric inclusion, MCP
   transport, and artifact database storage out of scope.
5. Add focused parser regressions in the existing claim/evidence benchmark
   test file, which is already enrolled in extracted checks.

### Review Contract

Acceptance criteria:
- Valid JSON generated by the existing result artifact bundle parses back into
  a `ClaimEvidenceResultArtifact` equivalent enough for the Markdown renderer
  and bundle writer to reuse.
- Malformed JSON and non-object JSON return a no-go artifact with explicit
  parser errors rather than raising.
- Missing or wrong-shaped thresholds, verdict, scores, agreement rows,
  stability rows, failure cases, or artifact error sections fail closed.
- Present verdict, `ok`, and `go_no_go` fields must match the
  parser-derived artifact state; contradictory saved artifacts fail closed.
- The parser does not call providers, read files, mutate the registry, or
  expose MCP behavior.

Affected surfaces: extracted claim/evidence benchmark package and existing
claim/evidence benchmark tests.

Risk areas: false-green saved benchmark evidence, malformed persisted JSON,
derived-field contradictions, and scope creep into live benchmark execution.

Reviewer rules triggered: R1, R2, R10, R12, R14.

## Mechanism

The package will expose a text loader and a decoded-mapping parser for the
saved result artifact JSON. The parser validates the same shape emitted by the
existing artifact mapper, converts nested rows back into the existing frozen
dataclasses, and returns a failed result artifact when validation errors are
found.

The `verdict.passed`, `verdict.failure_reasons`, `ok`, and `go_no_go` fields
remain derived from the dataclass state. If a saved payload includes those
fields and they disagree with the recomputed result, the parser reports the
contradiction as an artifact error instead of trusting the saved scalar.

## Intentional

- No file-path CLI is included here. The #1509 CLI already writes result files;
  this slice adds the package contract needed before a later operator command
  can validate or re-render a saved result file.
- No live provider adapter is introduced. Provider execution still requires
  model credentials, final labeled fixtures, and an explicit live-run slice.
- No verifier rubric or MCP inclusion is added. The slot remains excluded
  until real benchmark outputs pass the committed thresholds.

## Deferred

- A file-level validation or re-render command for saved result artifacts.
- Concrete provider adapters and live prompt execution.
- Batch model-run orchestration that calls real models and records response
  files.
- Operator labeling workflow for the final 40-row fixture.
- Verifier rubric inclusion and MCP exposure only if benchmark thresholds pass.

Parked hardening: none.
