# PR-Deflection-Proxy-Snapshot-Contract

## Why this slice exists

#1808 generated the in-repo `portfolio-ui` TypeScript result-page contract from
the backend `snapshot_projection`, but the Content Ops proxy still carries a
hand-written runtime projection in
`portfolio-ui/api/content-ops/deflection/atlas-report.js`. That copy is already
behind the backend contract: it omits `non_repeat_ticket_count`, per-question
`ticket_count`, and the `top_blind_spots` result-page section.

Root cause: the proxy validates and constructs its public snapshot envelope from
a manually maintained field list instead of the same backend contract metadata
that drives the generated frontend types.

This change fixes the root for the in-repo proxy surface by generating committed
runtime snapshot contract metadata from the backend projection and making the
proxy construct its public snapshot from those generated allowlists.

The diff is over the 400 LOC soft cap because the slice has three inseparable
parts: the generator change, the committed generated JS artifact, and the
runtime/proxy tests that prove the generated allowlists are load-bearing. Splitting
those would create a PR where the generated contract exists but is not enforced,
or enforcement exists without the generator gate.

## Scope (this PR)

Ownership lane: deflection/full-report-actionability
Slice phase: Production hardening

1. Generate a JavaScript runtime contract module beside the Content Ops proxy
   from `deflection_report_model_contract_shape().snapshot_projection`.
2. Update `projectSnapshot()` to construct `summary`, `top_questions`, and
   `top_blind_spots` from that generated metadata instead of a stale local copy.
3. Enroll the generated JS contract in the existing generator check and workflow
   path filters.
4. Add tests proving stale generated metadata fails, the proxy requires the
   result-page shape, optional summary fields stay optional, and non-allowlisted
   fields are stripped.

## Mechanism

`scripts/generate_deflection_frontend_contract_types.py` now renders both the
existing TypeScript type file and a new JS runtime metadata module for the
Content Ops proxy. The `--check` mode reads both generated outputs and fails if
either one is missing or stale.

`portfolio-ui/api/content-ops/deflection/atlas-report.js` imports the generated
field tuples and uses them as the allowlist for constructing the public snapshot.
Local coercers still own value validation because JavaScript needs runtime type
checks, but every constructed field must be listed by the generated contract
metadata.

## Intentional

- This does not touch `canfieldjuan/atlas-portfolio`; that remains a separate
  repo and a separate handoff unless the operator explicitly reassigns it.
- The generator name stays unchanged to avoid churn in the existing CI and test
  references; the script now owns frontend contract artifacts for both the React
  client and the in-repo API proxy.
- The proxy validates the result-page subset (`summary`, `top_questions`,
  `top_blind_spots`), not the paid-report-only `locked_questions` or `teaser`
  sections, because the proxy endpoint only returns the hosted result snapshot
  before artifact unlock.
