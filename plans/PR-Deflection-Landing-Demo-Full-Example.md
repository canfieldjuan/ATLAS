# PR-Deflection-Landing-Demo-Full-Example

## Why this slice exists

atlas-portfolio #389 now derives the public landing demo from the ATLAS-generated
report/snapshot examples, which is the right contract shape. The richer
producer-derived demo exposed that ATLAS #1858's synthetic input is still too
thin for the landing's configured preview: it has exactly five questions for a
`top_n=5` Snapshot, so `locked_questions` is empty; its proven/resolution-backed
cohorts have no reopened or negative-CSAT outcome risk, so
`already_covered_still_recurring` is empty; and the generated example does not
exercise the fuller mix the landing needs before the downstream hardening slice.

Root cause: the canonical synthetic input models high volume but not enough
support-queue variety. It proves count/evidence coherence, but it does not
include producer-truthful examples for locked backlog rows or "already covered
but still recurring" rows.

This PR fixes that root at the upstream producer fixture. It enriches the frozen
synthetic support-ticket cohorts, regenerates the paired report/snapshot
examples from the real producer, and pins the fuller shape in the generator
tests so the downstream portfolio demo can remain generated rather than
hand-authored.

This slice is over the normal LOC target because the generated report example is
the artifact under test. Splitting the JSON refresh away from the fixture/test
change would leave the producer and consumer contract temporarily inconsistent.

## Scope (this PR)

Ownership lane: deflection/landing-demo-contract-derived
Slice phase: Production hardening

1. Add synthetic cohorts that produce more than five ranked questions, including
   additional resolution-backed questions and one resolution-backed cohort with
   reopened/negative-CSAT risk.
2. Regenerate the paired report and Snapshot example JSON from the producer and
   extend the focused tests to require locked questions, already-covered rows,
   outcome diagnostics, and coherent evidence counts.

### Review Contract

Acceptance criteria:

- The generated example remains deterministic, synthetic-only, and coherent:
  no `source_weight`, unique `synthetic-*` source IDs, and count/proof/evidence
  surfaces agree.
- The generated Snapshot has at least one `locked_questions` row while keeping
  the public Snapshot free of question text in that locked projection.
- The generated report model has a non-empty `already_covered_still_recurring`
  section produced by real scoped resolution evidence plus reopened/negative
  outcome risk.
- The generated report model has a non-empty `outcome_diagnostics` section, and
  the Snapshot still includes blind spots for unresolved repeated questions.
- The example files are regenerated by
  `scripts/generate_deflection_snapshot_example.py`; no downstream
  atlas-portfolio constants are hand-authored here.

Affected surfaces:

- ATLAS docs/frontend generated report and Snapshot examples.
- atlas-portfolio #389 contract generator input.

Risk areas:

- Public-demo PII boundary: all new rows must stay fabricated and scrub-safe.
- Count/evidence credibility: extra rows must not reintroduce weighted-count
  contradictions.
- Review fixture bloat: the demo should get richer without returning to the
  1,140-row generated artifact.

## Mechanism

`synthetic_support_ticket_rows()` already expands `_SYNTHETIC_REPEAT_COHORTS`
into deterministic fabricated Zendesk-like rows. This slice keeps that mechanism
and only changes the cohort table:

- add a resolution-backed "saved search notification" cohort with reopened and
  negative-CSAT rows, which the existing report builder classifies as
  `Already covered but still recurring`;
- add another resolution-backed cohort below the top-five cutoff so the Snapshot
  emits locked backlog rows through the real `build_deflection_snapshot`
  projection;
- tune the existing counts enough that the top-five Snapshot contains a mix of
  proven and unresolved questions while the full report has more than five
  ranked rows.

The generator then rebuilds
`docs/frontend/content_ops_faq_deflection_report_example.json` and
`docs/frontend/content_ops_faq_deflection_snapshot_example.json`. Tests assert
the generated artifacts match the producer output and the key landing-demo
sections are non-empty.

The atlas-intel UI consumer test now derives its proven/no-proven count
expectations from the generated example instead of hardcoding the previous
fixture counts, so a future regenerated artifact exercises the consumer rather
than leaving a stale assertion behind.

## Intentional

- This stays at a moderate real-row volume rather than returning to the 1,140-row
  artifact from #1858's first pass. The target is "full enough to exercise the
  landing," not maximum marketing volume.
- The change does not add a synthetic `source_weight` shortcut; every count is
  backed by a real fabricated row and source ID.
- The portfolio rendering bug from #389 ("Top Proven Resolutions" labeling
  unresolved rows) is not fixed here; this PR only publishes the richer upstream
  generated artifact that the portfolio follow-up consumes.
